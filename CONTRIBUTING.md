# 🤝 Contributing Guide

본 프로젝트에 기여해주셔서 감사합니다.  
이 문서는 협업 과정에서의 일관성과 코드 품질을 유지하기 위한 가이드입니다.

---

## 📌 1. 브랜치 전략

다음과 같은 Git 브랜치 전략을 사용합니다.

- `main` : 배포 가능한 안정 버전
- `develop` : 개발 통합 브랜치
- `feature/*` : 기능 개발
- `fix/*` : 버그 수정
- `hotfix/*` : 긴급 수정

### ✔ 브랜치 네이밍 규칙

```

feature/기능명
fix/버그내용

```

예시:

```

feature/chat-ui
feature/rag-pipeline
fix/login-error

```

---

## 📌 2. 작업 프로세스

1. `develop` 브랜치에서 새 브랜치 생성
2. 기능 개발 진행
3. 커밋 후 원격 저장소에 push
4. Pull Request 생성
5. 코드 리뷰 후 `develop`에 merge

---

## 📌 3. 커밋 메시지 규칙

다음 형식을 따릅니다.

```

타입: 간단한 설명

```

### ✔ 타입 목록

- `feat` : 기능 추가
- `fix` : 버그 수정
- `docs` : 문서 수정
- `style` : 코드 스타일 변경
- `refactor` : 리팩토링
- `test` : 테스트 코드
- `chore` : 기타 변경

### ✔ 예시

```

feat: RAG pipeline 1차 구현
fix: 채팅 입력 버그 수정
docs: README 업데이트

```

---

## 📌 4. Pull Request 규칙

PR 작성 시 아래 내용을 포함해주세요.

### ✔ 필수 작성 항목

- 변경 내용 요약
- 작업 목적
- 주요 변경 사항
- 테스트 여부

### ✔ PR 제목 예시

```

[Feature] RAG pipeline 1차 구현
[Fix] 채팅 입력 오류 수정

```

---

## 📌 5. 코드 스타일

- 일관된 코드 스타일 유지
- 의미 있는 변수명 사용
- 불필요한 주석 지양 (필요한 경우 명확하게 작성)
- 함수는 하나의 책임만 갖도록 설계

---

## 📌 6. 협업 원칙

- 기능 단위로 작업하고 작은 단위로 PR 생성
- 코드 리뷰는 최소 1인 이상 승인 후 merge
- 주요 구조 변경 시 팀원과 사전 논의

---

## 📌 7. 이슈 관리

- 작업 전 Issue 생성
- Issue 기반 브랜치 생성 권장

예시:

```

#12 RAG 검색 정확도 개선

```

---

## 📌 8. 프로젝트 방향성

본 프로젝트는 단순 기능 구현이 아닌  
**사용자의 문제 해결을 중심으로 한 시스템 설계**를 목표로 합니다.

- 정보 제공 → 행동 유도 구조 설계
- RAG 기반 검색 정확도 개선
- 사용자 경험 중심 UX 설계
